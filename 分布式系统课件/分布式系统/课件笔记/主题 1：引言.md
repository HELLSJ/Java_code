## 计算机系统的演进

### 1945-1985 年

此时的计算机体积庞大、价格昂贵，均为独立运行。

### 20 世纪 80 年代中期至今

重大技术进步开始重塑计算领域：

- **高性能微处理器**：从 8 位 CPU 逐步过渡到 16 位、32 位和 64 位。多核 CPU 以极低的成本提供了强大的计算能力，如今一台普通机器的计算能力堪比三四十年前的大型主机，而成本仅为当时的千分之一左右。
- ![[Pasted image 20251112153609.png]]
- **高速计算机网络**：
    - 局域网（LANs）：可连接一栋建筑内的数千台设备，微秒级完成少量信息传输，大数据量传输速率可达每秒数十亿比特。
    - 广域网（WANs）：连接全球数亿台设备，传输速率从每秒数万比特到每秒数亿比特不等，甚至更高。
- **小型化与易用性提升**：
    - 智能手机：功能强大、传感器丰富且支持网络连接，配备多核 CPU 和大容量内存，属于完整的计算机设备，同时具备网络通信能力。
    - 纳米计算机：小巧廉价的单板计算机，尺寸通常与信用卡相当，性能接近桌面计算机，典型代表为树莓派（Raspberry Pi）。
- **数字化程度不断提高**：随着社会数字化进程推进，人们愈发意识到计算机的广泛应用 —— 它们常嵌入汽车、飞机、建筑、桥梁、电网等各类系统中。这些系统由联网计算机进行监控和控制，但这种认知往往在系统遭遇黑客攻击时才会凸显。例如 2021 年，美国一条输油管道因勒索软件攻击陷入瘫痪，该系统整合了传感器、执行器、控制器、嵌入式计算机和服务器等多种组件。事实上，输油管道等关键基础设施均由联网计算机系统负责监控和控制，这一点鲜为人知。

## 分布式系统的核心特征

### 规模

联网计算机系统的规模可从少数几台设备到数百万台计算机不等。

### 互联网络

- 互联网络可为有线、无线或两者结合的形式。这类系统通常具有高度动态性，计算机可随时加入或退出，底层网络的拓扑结构和性能也可能持续变化。
- 如今几乎所有计算机系统都具备联网功能，且多数联网计算机系统可通过互联网在全球范围内访问，其研究和理解难度较大。

## 集中式、去中心化与分布式的区别

三种网络计算机系统的组织形式不同，节点代表计算机系统，边代表两个节点间的通信链路：
![[Pasted image 20251112153731.png]]
### 集中式系统

单个中心节点连接多个外围节点，所有通信和控制都通过中心点进行。若中心节点故障，整个系统将陷入瘫痪。
### 去中心化系统

多个中心节点各自连接一组外围节点，中心节点之间存在部分连接。系统存在多个中心点，每个中心点负责网络的一个子集。单个中心节点故障不一定导致整个系统崩溃，比集中式系统更具韧性，但仍存在关键故障点。
### 分布式系统

多个节点以网状结构互联，无明确中心点。每个节点与多个其他节点连接，数据传输可通过多条路径进行。因无单点故障风险，系统具备高度韧性，适用于对可靠性和冗余性要求较高的大规模网络。

### 过渡边界

去中心化系统何时会转变为分布式系统？
- 向去中心化系统的节点间添加 k（k>0）条连接
- 随着连通性增强和对中心节点依赖降低，网络的韧性和容错能力会逐步提升，最终实现从去中心化到分布式的过渡。

## 分布式系统的两种实现视角

### 整合视角

需将现有（联网）计算机系统相互连接，通常是为了让系统中的服务能够供未预想到的用户和应用程序访问。例如，企业内部常需将金融服务与项目管理服务整合。

### 扩展视角

通过添加更多计算机扩展现有系统，这是分布式系统领域最常见的视角。扩展的目的包括将资源部署在需求集中的位置、提升可靠性（某台计算机故障时，其他计算机可接管任务），以及为远程用户和应用程序提供服务（如通过 Web 界面或智能手机应用程序）。两种视角的界限并非绝对清晰。

## 去中心化与分布式系统的定义

### 去中心化系统

进程和资源必须分布在多台计算机上，主要与联网计算机系统的整合视角相关。其产生是为了连接不同系统，但可能受行政边界限制。例如，人工智能领域的许多应用需要海量数据训练预测模型，通常是将数据传输到高性能计算机进行训练；但当数据需留在组织内部时，就需要将训练过程迁移到数据所在位置。联邦学习（Federated Learning）就是去中心化系统的典型应用，其进程和资源的分布由行政政策决定。

### 分布式系统

进程和资源充分分布在多台计算机上，主要与联网计算机系统的扩展视角相关。例如谷歌邮箱（Google Mail）：用户通过 Web 界面或邮件客户端登录使用，配置的接收服务器（[imap.gmail.com](https://imap.gmail.com/)）和发送服务器（[smtp.gmail.com](https://smtp.gmail.com/)）看似是处理所有邮件的核心，但截至 2022 年谷歌邮箱用户约 20 亿，每年邮件发送量超 3000 亿封（约每秒 1 万封），实际由大量计算机共同组成分布式系统支撑。该系统可根据用户数量动态扩展或缩减，确保可扩展性和容错性，同时对用户隐藏分布细节，保持高度透明。

### 小型分布式系统案例

网络附加存储（NAS）设备：家用 NAS 通常有 2-4 个内置硬盘插槽，作为文件服务器通过（无线）网络供授权设备访问，提供共享存储、自动备份、媒体流等服务。用户可从本地设备（如笔记本电脑）便捷访问共享文件，且文件可被其他用户直接访问，存储位置和方式对用户透明，属于典型的小型分布式系统。

## 常见误解

### 误解 1：集中式方案无法扩展

实际应区分逻辑集中式和物理集中式：

- 逻辑集中式方案可通过高度可扩展的分布式方式实现。
- 例如域名系统（DNS）的根节点：逻辑上是单一的名称层级根，但物理上分布在全球多个服务器集群中（13 个根服务器，每个由大量计算机组成），具备高可用性和韧性，并非单点故障。

### 误解 2：集中式方案存在单点故障

并非绝对（如 DNS 根节点）。单点故障点通常更易于管理和增强鲁棒性。集中式方案并非天生劣势，逻辑集中式方案有时更易于管理和高效运行，关键在于对潜在故障点进行强化。许多基于云的集中式方案已被证明具有极高的可扩展性和鲁棒性。

### 关键结论

关于可扩展性、容错性、安全性等方面存在诸多无根据的误解，需培养理解分布式系统的能力，客观评判这些观点。集中式、去中心化和分布式系统之间无严格界限，可相互转化。



## 理解分布式系统的视角

分布式系统复杂且存在大量显式和隐式依赖，需从多个结构化视角进行理解：

### 架构视角

了解分布式系统的整体结构，分析常见的组织形式和风格，掌握系统各部分的交互和依赖关系，是系统设计的基础。

### 进程视角

进程是分布式系统的核心组件，理解线程、客户端、服务器等不同类型的进程，是掌握分布式系统基本运行机制的关键。

- 通信视角：探索进程间的数据交换方式，是确保分布式系统功能和性能的核心，直接影响数据传输的有效性和效率。
- 协调视角：关注进程间的协同机制，包括共享资源访问管理、全局时钟缺失的补偿等，确保分布式进程无缝协作。
- 命名视角：研究有效的命名方案，明确名称如何解析为访问点，是分布式系统中资源定位和使用的基础。
- 一致性与复制视角：分析资源复制带来的一致性挑战，权衡一致性、复制和性能之间的关系，是维护系统完整性的关键。
- 容错视角：理解系统如何掩盖故障并实现恢复，是确保分布式系统可靠性的重要领域，也是最具挑战性的视角之一。
- 安全视角：聚焦资源的授权访问，包括机密性、完整性保障等，本课程后续可能不单独深入讨论。
## 分布式系统的总体设计目标

### 支持资源共享

- 成本效益：共享大型存储系统比个人单独购置和维护更经济。
- 协作便利：连接用户和资源，促进信息共享与协作，例如互联网的文件共享协议、OneDrive、iCloud Drive、谷歌云端硬盘（Google Drive）、Dropbox 等云存储服务。
- 典型案例：
    - 群件（Groupware）：支持视频通话、文档共享等协作功能，在新冠疫情期间远程办公场景中发挥重要作用。
    - 对等网络（P2P）：如 BitTorrent，用户可相互共享文件片段，高效分发大型文件（如电影、软件更新），降低单台服务器负载。
    - 电子邮件系统：如 Gmail、Outlook，由第三方服务提供商管理存储、垃圾邮件过滤和安全保障，无需组织自行维护服务器。
    - 内容分发网络（CDNs）：如 Cloudflare、Akamai，将网站内容分布在全球多个服务器，用户从最近的服务器获取内容，提升加载速度和可靠性。

### 分布透明性

- 定义：隐藏进程和资源在物理上的分布式特性（可能跨远距离），让系统对用户和应用程序呈现为单一、连贯的整体，交互过程无缝直观。
- 实现方式：中间件（Middleware）是关键组件，作为应用程序和底层操作系统之间的层，确保用户在任何位置看到统一的接口。
- 类型：

| 透明性类型  | 描述                     |
| ------ | ---------------------- |
| 访问透明性  | 隐藏数据表示和对象访问方式的差异       |
| 位置透明性  | 隐藏对象的物理位置              |
| 重定位透明性 | 隐藏对象在使用过程中可能迁移到其他位置的事实 |
| 迁移透明性  | 隐藏对象可能迁移到其他位置的事实       |
| 复制透明性  | 隐藏对象被复制的事实             |
| 并发透明性  | 隐藏对象可能被多个独立用户共享的事实     |
| 故障透明性  | 隐藏对象的故障和恢复过程           |

- 透明性的局限性：完全实现分布透明性不切实际，例如通信延迟无法隐藏、网络和节点故障难以完全掩盖（无法区分计算机缓慢和故障）、服务器崩溃前的操作执行状态无法确认。过度追求透明性会影响性能（如副本同步、写操作磁盘刷新）。
    
- 适度暴露分布特性的优势：支持基于位置的服务（如查找附近好友）、适配不同时区用户需求、帮助用户理解系统行为（如告知服务器可能故障）。
    
- 结论：需平衡透明性水平与系统及用户需求，追求部分透明性或明确暴露特定分布特性，以提升用户体验和系统性能。
    

### 开放性

- 定义：系统组件可被其他系统轻松使用或整合，分布式开放系统本身通常由来自不同来源的组件组成。组件需遵循标准规则，通过接口定义语言（IDL）描述提供的服务，明确可用函数名称。
- 核心要求：
    - 互操作性：不同厂商的系统或组件可基于共同标准共存并协作。
    - 可移植性：为分布式系统 A 开发的应用程序可在实现相同接口的分布式系统 B 上无修改运行。
    - 可扩展性：易于添加不同操作系统的组件或替换整个文件系统（如浏览器插件、WordPress 网站插件）。
- 实现方式：
    - 策略：指导系统行为的规则、指南和目标（如客户端缓存数据的一致性级别、下载代码的允许操作、带宽变化时的服务质量调整、通信保密性要求）。
    - 机制：执行策略的方法、流程和工具（如动态缓存策略设置、移动代码信任级别支持、数据流服务质量参数调整、加密算法提供）。
- 权衡：严格分离策略和机制理论上是理想方案，但会导致机制集合庞大、配置参数繁多（如 Firefox 浏览器有数百个配置参数），增加系统复杂度。实际中常采用合理默认值或自配置系统（根据使用情况动态调整参数），也可通过硬编码策略降低复杂度，但会牺牲灵活性。

### 可靠性

- 定义：系统按预期运行的程度。分布式系统的可靠性更复杂，因可能出现部分故障（某组件故障但系统整体仍能满足预期），且需默认随时可能发生部分故障，核心目标是掩盖故障及恢复过程，实现容错。
- 组件依赖：组件 C 依赖组件 C*，指 C 的行为正确性取决于 C * 的行为正确性（组件包括进程或通道）。
- 核心要求：
    
| 要求类型 | 描述                                                    |
| ---- | ----------------------------------------------------- |
| 可用性  | 系统随时可立即使用的概率，即某一时刻系统正常运行的可能性                          |
| 可靠性  | 系统在一段时间内连续无故障运行的特性（与可用性的区别：可用性关注 “某一时刻”，可靠性关注 “一段时间”） |
| 安全性  | 系统临时故障时不发生灾难性事件（如核电厂、航天控制等过程控制系统对安全性要求极高）             |
| 可维护性 | 故障系统易于修复的程度，高可维护性通常伴随高可用性（如自动故障检测和修复）                 |
- 关键指标：
    - 平均无故障时间（MTTF）：组件故障前的平均运行时间。
    - 平均修复时间（MTTR）：组件故障后的平均修复时间。
    - 平均故障间隔时间（MTBF）：MTTF + MTTR。
- 故障相关概念：

| 术语          | 描述          | 示例    |
| ----------- | ----------- | ----- |
| 故障（Failure） | 组件未达到规格要求   | 程序崩溃  |
| 错误（Error）   | 可能导致故障的组件缺陷 | 编程漏洞  |
| 缺陷（Fault）   | 错误的原因       | 程序员疏忽 |

- 故障类型：
    - 瞬时故障：发生一次后消失，重试操作可能成功（如鸟穿过微波发射器波束导致网络比特丢失）。
    - 间歇性故障：反复出现和消失（如连接器接触不良）。
    - 永久性故障：需更换故障组件才能解决（如芯片烧毁、软件漏洞、磁盘磁头崩溃）。
- 故障处理方式：

| 术语   | 描述                  | 示例                    |
| ---- | ------------------- | --------------------- |
| 故障预防 | 防止故障发生              | 不雇佣疏忽的程序员             |
| 容错   | 构建组件以掩盖故障           | 由两名独立程序员开发同一组件        |
| 故障消除 | 减少故障的存在、数量或严重程度     | 解雇疏忽的程序员              |
| 故障预测 | 估计当前故障存在情况、未来发生率及影响 | 评估招聘人员在避免雇佣疏忽程序员方面的表现 |


### 安全性

- 核心关联：可靠的系统必须具备安全性（尤其是机密性和完整性），不安全的分布式系统无法称为可靠。
    
- 核心要求：
    
    - 机密性：信息仅向授权方披露。
    - 完整性：确保系统资产的修改只能通过授权方式进行。
- 关键概念：
    
    - 认证（Authentication）：验证声称身份的正确性。
    - 授权（Authorization）：确认已认证实体是否具备适当的访问权限。
    - 信任（Trust）：一个实体确信另一个实体将按特定预期执行特定操作。
- 实现基础：基于密钥的加密和解密。密钥 K 可视为对数据进行操作的函数，K (data) 表示使用密钥 K 加密 / 解密数据。
    
- 加密体系：
    
    - 对称加密体系：加密和解密使用同一密钥，加密密钥 EK (data) 和解密密钥 DK (data) 满足 data = DK (EK (data))，且 DK=EK，密钥需严格保密。
    - 非对称加密体系：区分公钥 PK (data) 和私钥 SK (data)，满足 data = SK_bob (PK_bob (data))。例如 Alice 向 Bob 发送签名消息：Alice 发送 [data, SK_alice (data)]，Bob 通过验证 data 是否等于 PK_alice (SK_alice (data)) 确认消息有效性。
- 安全哈希：
    
    - 特性：哈希函数 H (data) 返回固定长度字符串；数据修改（data→data*）会导致哈希值完全不同；已知哈希值无法计算出原始数据。
    - 实际应用：数字签名通常结合哈希函数，Alice 向 Bob 发送签名消息的流程为：Alice 计算 $sgn=SK_{alice} (H (data))$，发送 \[data, sgn]；Bob 验证 H (data) 是否等于 PK_alice (sgn)。
- 分布式系统中的加密应用：
    
    - 安全通道：确保双方安全通信（如 HTTPS 用于安全 Web 访问）。
    - 区块链：由哈希数据组成的区块链，每个区块 $B_i$ 包含其数据的哈希值，且 $B_i$ 的哈希值包含在 $B_{i+1}$ 中，修改任一区块需修改后续所有区块。
    - 访问权限委托：Alice 将权限委托给 Bob，Bob 再委托给 David，系统可直接验证 David 的授权，无需每次查询 Alice。
    - 多方计算：多方联合计算某值而不泄露各自数据（如安全计票，不公开个人投票）。

### 可扩展性

- 定义：系统应对增长（用户数量、数据量、地理范围等）的能力，许多分布式系统开发者对 “可扩展” 的表述缺乏明确依据。
    
- 核心维度：
    
    - 规模可扩展性：支持更多用户或进程。
    - 地理可扩展性：节点间最大距离增加。
    - 管理可扩展性：支持更多管理域。
- 现状：多数系统仅在一定程度上考虑规模可扩展性（如通过多台并行服务器实现），地理可扩展性和管理可扩展性仍是主要挑战。
    
- 规模可扩展性问题根源（集中式方案）：
    
    - 计算能力有限（受 CPU 限制）。
    - 存储能力有限（含 CPU 与磁盘间的传输速率限制）。
    - 网络带宽限制（用户与集中式服务间的网络瓶颈）。
- 形式化分析（集中式服务排队模型）：
    
    - 假设：队列容量无限，请求到达率 λ 不受队列长度或处理状态影响；服务处理能力为 μ（每秒处理请求数）。
    - 关键公式：
        - 利用率 U=λ/μ（服务忙时占比）。
        - 系统中平均请求数 N=U/(1-U)。
        - 平均吞吐量 X=λ。
        - 响应时间 R=N/X=1/(μ(1-U))=S/(1-U)（S 为服务时间）。
    - 结论：U 较小时，响应时间接近服务时间；U 趋近于 1 时，系统陷入停滞，解决方案是缩短服务时间 S。
- 地理可扩展性问题：
    
    - 局域网（LAN）到广域网（WAN）的迁移挑战：分布式系统常假设客户端 - 服务器同步交互，WAN 的高延迟会导致该模式不可行。
    - WAN 链路可靠性低：如 LAN 中的流式视频无法直接迁移到 WAN。
    - 缺乏多点通信：无法部署简单的搜索广播，需开发独立的命名和目录服务（自身也存在可扩展性问题）。
- 管理可扩展性问题：
    
    - 核心矛盾：不同管理域在使用规则（含付费）、管理方式和安全策略上的冲突。
    - 示例：计算网格（跨域共享昂贵资源）、共享射电望远镜（大规模共享传感器网络的控制和管理）。
    - 例外：部分对等网络（如 BitTorrent 文件共享、早期 Skype 对等电话、Spotify 对等辅助音频流），其协作主体是终端用户而非管理实体，规避了管理域冲突。
- 可扩展性优化技术：
    
    - 升级扩展（Scaling Up）：提升单台服务器性能（如增加内存、升级 CPU、更换网络模块）。
    - 横向扩展（Scaling Out）：部署更多机器，核心技术包括：
        1. 隐藏通信延迟：尽量避免等待远程服务响应，采用异步通信、独立响应处理程序（局限性：并非所有应用都适用）。
        2. 计算迁移到客户端：将部分计算任务（如表单验证）交由客户端处理，减少服务器负载和数据传输量（示例：客户端本地验证表单数据后，仅向服务器提交有效数据，服务器直接处理无需再次验证）。
        3. 数据和计算分区：将数据和计算任务分布到多台机器（如 DNS 的域名空间分层分区、WWW 的文档分布式存储）。
        4. 复制和缓存：在不同机器上存储数据副本（如复制文件服务器、镜像网站、Web 缓存、文件缓存），提升访问速度和可靠性。
- 复制和缓存的挑战：多副本会导致一致性问题（修改一个副本会使该副本与其他副本不一致）。全局同步所有副本会限制大规模扩展，实际中需根据应用场景容忍一定程度的不一致。

## 分布式系统的分类

根据开发目标和用途，分布式系统主要分为以下类别：

### 高性能计算

- 并行计算：高性能分布式计算的起源，分为共享内存多处理器 / 多核系统（所有 CPU 访问同一物理内存，编程简便）和分布式内存多计算机系统（每个 CPU 有私有内存，通过互联网络通信）。
- 分布式共享内存系统：试图在多计算机系统上实现共享内存模型（如通过虚拟内存技术将不同处理器的内存页映射到单一虚拟地址空间），但性能无法与多处理器竞争，已被广泛弃用。
- 集群计算（Cluster Computing）：由一组相似的计算节点通过高速网络互联组成，通常搭配局域网用于节点控制，所有节点运行相同操作系统。因个人计算机和工作站的性价比显著提升，集群计算成为构建超级计算机的经济可行方案，适用于并行编程（单一计算密集型程序在多台机器上同时运行）。大型集群可包含超过 10 万个 CPU（每个 CPU 含 8-16 个核），配备高速互联网络、管理网络和专用高性能文件系统 / 数据库，操作系统向轻量级内核发展，或采用多核架构（全功能操作系统与轻量级内核结合）。
- 网格计算（Grid Computing）：去中心化系统，由多个不同管理域的计算机系统组成联盟，硬件、软件和网络技术可能存在显著差异，通过虚拟组织实现资源分配和授权协作。
- 网格计算架构（从底层到上层）：
    1. 设施层（Fabric）：提供本地资源接口（如状态查询、能力查询、锁定）。
    2. 连接层（Connectivity）：提供通信 / 事务协议（如数据传输、认证协议）。
    3. 资源层（Resource）：管理单个资源（如创建进程、读取数据）。
    4. 集合层（Collective）：处理多资源访问（如资源发现、调度、复制）。
    5. 应用层（Application）：包含特定组织的网格应用程序。

### 分布式普适系统

下一代分布式系统，节点小巧、移动且常嵌入更大的系统中，自然融入用户环境，主要分为三个重叠亚型：

- 普适计算系统（Ubiquitous Computing Systems）：核心要素：设备联网且分布式透明访问；用户与设备的交互高度不显眼；系统感知用户上下文以优化交互；设备自主运行（无需人工干预）；系统整体可处理多种动态行为和交互。
    
- 移动计算系统（Mobile Computing Systems）：显著特征：移动设备种类繁多（智能手机、平板电脑、GPS 设备、遥控器、有源徽章）；设备位置动态变化（导致本地服务、可达性等变化，核心需求为服务发现）；通信稳定性面临挑战。当前移动计算的主流模式是移动设备连接固定服务器，依托云服务，主要分为移动云计算（MCC）和移动边缘计算（MEC）。MEC 适用于低延迟、高计算需求的场景（如增强现实、交互式游戏、实时运动监测、健康应用），将计算任务部署在靠近移动设备的边缘数据中心，解决云服务的高延迟问题。

- 传感器网络（Sensor Networks）：特征：传感器节点数量多（数十到数千个）、结构简单（内存、计算和通信能力有限）、常采用电池供电（或无电池）。传感器网络可视为分布式数据库，数据处理有两种极端模式：传感器直接将数据发送到操作员站点；传感器本地处理数据，仅向操作员站点返回查询结果。

## 分布式系统开发的常见陷阱

许多分布式系统因前期错误假设导致不必要的复杂性，后续需频繁修补。常见的错误（且常隐藏）假设包括：

- 网络是可靠的
- 网络是安全的
- 网络是同构的
- 网络拓扑不会变化
- 延迟为零
- 带宽无限
- 传输成本为零
- 存在单一管理员

这些假设与分布式系统的独特属性相关，而非分布式应用开发中通常不会遇到的问题。
